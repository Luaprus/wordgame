# 文字关卡制作通用工作流

这份文档用于给剑关、手套关等文字关卡制作人员参考。重点不是某一关的具体解法，而是通用制作流程：如何复刻地图、锁定坐标、实现交互、组织状态、接入动画和测试。

## 核心原则

先不要急着做“能玩”，先做“看起来完全对”。

稳定顺序是：

```text
地图准确
坐标锁定
交互清晰
状态明确
动画补上
测试兜底
```

不要一开始就把地图、交互、动画和流程混在一起做。地图不准时，后续所有交互坐标都会偏；交互坐标不准时，流程状态也会乱。

## 1. 把截图拆成固定网格

先把画面统一理解成固定网格，例如 `18 x 32`。

每个字、标点、空格都算一个格子。需要先记录：

- 玩家初始位置
- 每个可见文字的位置
- 障碍物位置
- 可交互文字位置
- 初始应该出现的说明文字
- 初始不应该出现、需要交互后才出现的文字

地图可以用固定长度字符串数组表达：

```gdscript
"  一句话文字      障障障障障      "
"  第二行文字      障障障障障      "
```

要求：

- 每一行长度必须一致。
- 空格也要算格子。
- 标点也占一格。
- 玩家如果嵌在文字句子里，那个格子通常作为玩家初始坐标，不再生成普通文字实体。

## 2. 先只做静态地图

第一版不要做交互、动画、流程，只追求截图一致。

检查内容：

- 字的位置一致
- 玩家位置一致
- 行列间距一致
- 标点一致
- 空白区域一致
- 障碍边缘一致
- 初始不该出现的文字没有提前出现

实现后截图对照原图，一个格一个格检查。地图不准时，不要用逻辑补救，直接改地图。

## 3. 用测试锁住关键坐标

每关至少写基础地图测试：

- 行数正确
- 每行长度正确
- 玩家初始坐标正确
- 关键文字坐标正确
- 关键障碍坐标正确
- 初始隐藏文字不存在
- 初始可交互对象存在

这样后续做动画、事件和状态切换时，不会无意中把地图弄偏。

## 4. 把交互当成地图事件

按 Space、走到某格、触发某动画，都不要写成分散的临时逻辑。建议统一成事件配置：

```gdscript
"某字": {
	"solid": true,
	"interact_text": "...",
	"interact_effect": _some_effect()
}
```

`effect` 负责描述交互后的地图变化，例如：

- 生成说明文字
- 删除某些字
- 生成新字
- 锁定玩家输入
- 解锁玩家输入
- 触发延迟事件
- 切换状态
- 切换关卡

这样关卡流程会更清楚，也方便写测试。

## 5. 说明文字也按地图文字处理

如果交互后左侧出现说明文字，它也应该是地图上的文字实体，而不是普通 UI 弹窗。

要求：

- 一个字占一格
- 固定坐标生成
- 可以设置是否有碰撞
- 出现后是否消失由关卡流程决定
- 玩家能不能穿过去要明确配置

这样才能保持文字地图的整体一致性。

## 6. 复杂流程拆成状态

如果剑关、手套关有动画或阶段变化，建议先把流程拆成明确状态：

```text
初始状态
交互后状态
动画播放中状态
动画结束状态
成功状态
失败状态
切换场景状态
```

每个状态都要写清楚：

- 玩家能不能移动
- 哪些字存在
- 哪些字可交互
- 哪些动画播放
- 下一步触发条件是什么

不要把所有流程塞进一个长函数里。

## 7. 先实现无动画版本

动画关卡尤其要先做逻辑正确：

- 触发条件正确
- 玩家锁定正确
- 字出现和消失正确
- 状态切换正确
- 成功或失败结果正确

这些确认后，再接动画表现。

否则容易出现“动画看起来播放了，但关卡状态其实错了”的问题。

## 8. 延迟事件统一管理

如果有关卡需要“等 1 秒后发生某事”“动画结束后发生某事”，不要随便在各处写 timer。

建议统一成类似结构：

```gdscript
pending_timed_effect
pending_timed_delay
```

主场景统一检查并执行。这样死亡、物体移动、动画后结算、延迟生成文字都可以用同一套机制。

## 9. 边界和切关放在主控

每个关卡只负责自己的地图和事件。串关逻辑放在主控里。

推荐规则：

- 关卡可配置是否开启边界
- 上、下、左边界阻挡
- 右边界触发 `next_level`
- 主控收到 `next_level` 后加载下一关
- 特殊关卡可以带入玩家位置或玩家状态

这样单关不会互相污染，后续调整关卡顺序也更容易。

## 10. 每关都写回归测试

不一定测所有视觉细节，但要测关键流程。

建议覆盖：

- 初始地图
- 初始玩家位置
- 关键交互
- 触发后文字出现
- 触发后状态变化
- 玩家是否被锁定或解锁
- 动画后结果
- 成功流程
- 失败流程
- 关卡切换

## 推荐工作流

```text
读需求和截图
拆成固定网格
写静态地图
截图对照修正
写初始地图测试
加 Space 或走格触发
加状态变化
加延迟事件
加动画表现
加切关
跑测试
```

## 注意事项

- 不要边摆地图边写复杂玩法。
- 不要用逻辑去弥补地图坐标错误。
- 不要让说明文字变成和地图无关的 UI。
- 不要把状态变化散落在多个临时判断里。
- 不要先做动画再补状态。
- 每个容易改坏的流程都要有测试。

## 可选扩展：文字状态变化

如果某些关卡需要文字变化、文字组合或多个同类文字同步变化，再额外设计文字状态规则。

剑关和手套关如果没有拆字、合字、同步组，可以不用这部分。
