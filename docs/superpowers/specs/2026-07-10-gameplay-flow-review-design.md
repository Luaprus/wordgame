# F033 自然语言玩法流程标注系统设计

## 1. 目标与范围

本设计新增 `F033：自然语言玩法流程标注系统`，用于收集、复查并交接三个待复刻关卡的完整玩法语义。它补足现有视频事件、截图语义和源码索引的粒度：审核者能够用中文记录玩家所见、所有可尝试操作、前置条件、结果、失败、重置、终局和人工判断。

本系统覆盖三个独立页面：

| 页面 | 关卡 ID | 任务边界 |
| --- | --- | --- |
| 剑关 | `sword` | 仅维护剑关的情境、分支、证据和备注 |
| 手套关 | `glove` | 仅维护手套关的情境、分支、证据和备注 |
| 头盔关 | `helmet` | 仅维护头盔关的情境、分支、证据和备注 |

总入口仅提供关卡分发、各关状态和导出入口。审核者进入某个关卡后，只能编辑该关数据。跨关规律只能以备注引用，不能直接改写另一关的数据。

本功能不实现游戏玩法、不替换既有截图或视频审核页面、不改动仍由组员处理的头盔基线导入数据。它只建立一层可追溯的玩法语义审核系统。

## 2. 核心原则

1. 不预设情境、操作或分支数量。审核者可以在任意位置新增情境、操作和结果，形成无限展开的流程树。
2. 每种可尝试操作都必须记录，包括无效操作、失败、重置和不推进主线的结果。
3. 同一操作在不同前置条件下结果不同，必须作为不同的条件分支记录，不能合并为模糊描述。
4. 审核者的中文原话永久保留。自动整理出的结构化字段是派生数据，不能覆盖原话。
5. 未确认的信息保持待补充或待复查；系统不能根据视频、截图或源码自行补猜后标为确认。
6. 只有已确认信息可以被后续游戏实现任务读取。

## 3. 审核单元与数据模型

系统的最小单位是“情境”，不是预先编号的卡片。一个情境描述玩家当前看到什么、处于什么状态、可做什么。情境可随时新增，稳定 ID 由系统创建而不是由审核者手填。

每个情境至少包含：

| 字段 | 说明 |
| --- | --- |
| 情境名称 | 审核者可读的中文名称 |
| 情境概述 | 当前发生了什么、玩家在哪里 |
| 玩家所见 | 可见文字、对象、地形、提示和状态变化 |
| 玩家目标或发现 | 当下可理解的目标、线索或限制；未知可留空 |
| 可操作项 | 任意数量的操作记录 |
| 情境备注 | 与整段情境相关的人工说明、疑问或复查提示 |
| 证据 | 视频时间、截图、原版实测或源码引用 |
| 复查状态 | 待补充、待复查、已确认或不纳入复刻 |

每个可操作项至少包含：操作名称、输入方式、目标文字或对象、前置条件、结果、下一情境或终局、操作备注和证据。无效、失败、重置也属于有效的“结果类型”，不得省略。

每个结果可以继续携带“结果备注”。备注可以挂在情境、操作或结果三级任意一层，并保留原话、填写人、填写时间、确认状态。备注在明确确认前不能作为实现事实。

## 4. 页面与交互

### 4.1 总入口

总入口显示剑关、手套关、头盔关三张关卡条目。每条目显示：负责人、已确认数量、待补充数量、待复查数量、最后修改时间和“进入本关”入口。总入口不允许编辑玩法内容。

### 4.2 关卡审核页

每个关卡页包含以下固定区域：

1. 情境列表：显示本关所有情境；可随时新增，并支持定位到缺漏项。
2. 当前情境编辑区：以中文输入情境概述、玩家所见、可操作项、结果、证据和备注。
3. 自动流程树：根据当前关的情境和结果连接生成；任何新增分支都会自动加入。
4. 缺漏与复查区：展示当前关所有未满足确认门槛的具体项目。
5. 本关验收与导出区：运行检查；只有通过后才允许导出“已确认玩法包”。

审核者不需要先决定情境数量。新增操作时，页面允许选择已有情境作为结果，创建新情境，或标为失败、重置、终局、保持当前情境。每个操作都明确记录其前置条件；“无前置条件”必须显式选择或填写。

## 5. 信息保真与状态流转

信息经历四层，且可以完整回溯：

1. 人工原话：审核者填写的中文描述和备注，永不被自动改写。
2. 结构化草稿：页面根据明确字段形成的情境、操作、条件、结果关系；仍可待补充或待复查。
3. 人工确认：确认人基于证据确认结构化内容与原话一致。
4. 可实现基线：仅导出已确认的内容，供 Codex 和开发者实现、写测试和验收。

状态定义：

| 状态 | 含义 | 是否允许进入实现基线 |
| --- | --- | --- |
| 待补充 | 描述、条件、结果或证据缺失 | 否 |
| 待复查 | 已有说法，但证据不足、存在冲突或需要第二人确认 | 否 |
| 已确认 | 原话、结构、证据和复查信息齐全且一致 | 是 |
| 不纳入复刻 | 明确无关、超出复刻范围或被判定为错误材料 | 否 |

“已确认”不表示原版所有可能性已被穷尽，只表示当前这一条具体记录已经由证据和审核人确认。新增分支会重新触发完整性检查。

## 6. 自动校验、错误处理与验收

每关独立运行校验，至少检查以下规则：

1. 每个已确认情境至少有一条证据。
2. 每个已确认操作都有明确条件（包括“无”）、结果和结果类型。
3. 结果为跳转时，必须指向存在的情境；结果为终局、失败、重置或保持时，必须明确类型。
4. 分支操作不能用“进入某处”“有变化”等模糊结果代替具体结果。
5. 有备注但未确认的内容不得被导入已确认玩法包。
6. 不纳入复刻的内容必须有排除原因。
7. 每关导出只包含该关已确认数据；另一关的数据不能被混入。

校验失败不覆盖人工输入，而是定位到对应情境、操作或结果，并列为待处理项。证据相互矛盾时，系统保留不同记录和备注，将状态置为待复查。原版无法验证时，保留“人工待定”并阻止其进入实现任务。

每关可以单独验收：所有校验通过、没有待补充或待复查的“已确认候选”项目、导出包可被机器读取、人工审核记录完整。导出包分为两个明确隔离的部分：`confirmed_flow` 只含可用于实现的已确认情境、操作、条件、结果与证据；`review_manifest` 保留人工原话、备注状态、验收结果和未纳入复刻清单，只能用于追溯，不能被实现任务当作玩法事实读取。

## 7. 模块边界与拟定文件

实现阶段将新增或修改以下范围，实际变更必须先登记进 `F033` 的 `allowed_files`：

| 模块 | 职责 | 规划路径 |
| --- | --- | --- |
| 玩法流 schema | 定义机器可校验的数据契约 | `harness/baselines/schema/gameplay_flow.schema.json` |
| 三关数据 | 保存各关审核数据和导出结果 | `harness/gameplay_flow/` |
| 审核页面 | 总入口及三个独立中文审核页 | `harness/gameplay_review/` |
| 构建工具 | 将页面数据构建为可审核页面 | `tools/build_gameplay_review_app.mjs` |
| 校验工具 | 检查完整性、引用和关卡隔离 | `tools/validate_gameplay_flow.mjs` |
| 导入工具 | 读取人工审核结果，保留原话与状态 | `tools/import_gameplay_review_result.mjs` |
| Harness 文档 | 功能、契约、测试矩阵、验收清单 | `harness/features.json`、`harness/contracts.md`、`harness/plan.md`、`harness/test_matrix.md`、`harness/acceptance.md` |

既有 `harness/manual_review/` 保持为视频和截图语义审核系统；新模块只通过证据引用关联它，不改写其审核结果。

## 8. 测试证据

`F033` 完成前必须由一键测试入口验证，至少包含：

1. schema 测试：合法情境、操作、条件、结果和备注可通过；缺少必填关系必须失败。
2. 关卡隔离测试：剑关、手套关、头盔关的数据不得交叉引用或交叉导出。
3. 状态门禁测试：待补充、待复查和不纳入复刻项不得进入已确认玩法包。
4. 分支完整性测试：每条已确认操作必须有可解析结果；跳转不能指向不存在情境。
5. 导入导出往返测试：人工原话、备注、状态、证据和结构在导入导出后不丢失。
6. 视觉烟雾测试：总入口和三关页可打开；中文标签可见；流程树、缺漏提示和备注区不重叠、不溢出。

所有通过证据必须记录到 `harness/progress.jsonl`。测试失败、存在未处理待复查项或导出包不完整时，`F033` 不得标记完成。

## 9. 非目标与待实施顺序

本功能不自动理解视频或源码中的未知玩法，不替代人工确认，也不直接修改 Godot 游戏实现。它先让人工用可扩展、可校验的方式把玩法语义交给开发流程。

实施顺序为：登记 `F033` 契约与验收条件，定义 schema 和空数据，构建中文审核页面，编写导入导出与校验，接入一键测试，生成三关初始审核包，最后由人工分发审核。
