# 游戏工程统一与分层设计

## 目标

将仓库内多个彼此分叉的 Godot 工程统一为一个正式游戏工程，按软件职责组织文件，消除“同一核心脚本存在多份副本”导致的合并冲突。

本次迁移只调整工程归属和资源路径，不改变任何关卡玩法、地图、动画时序或人工验收结论。

## 当前问题

仓库同时存在以下 Godot 工程：

- 根目录工程：旧主菜单、第一章资源和一套核心脚本。
- `newgame/`：手套关、头盔关、完整试玩入口和测试最齐全的工程。
- `test game/`：`newgame/` 的较小副本，核心脚本已经与 `newgame/` 重合。
- `剑流程/`：独立的剑关工程。

多个目录都包含 `project.godot`、`main.gd` 和 `grid_world.gd`。同一功能可能被提交到不同副本，例如视觉效果队列曾出现主入口已调用、实际运行的世界脚本却没有实现的情况。

## 目标目录

迁移完成后，仓库根目录只保留一个正式 Godot 入口：`project.godot`。

```text
app/                    启动、主菜单、场景切换
core/                   格子世界、规则、文字实体、关卡加载
gameplay/               移动、镜头、通用互动和演出调度
features/               各关卡的专属规则、效果和流程控制
    glove/
    helmet/
    sword/
content/                关卡配置、地图、场景和关卡数据
    levels/
    scenes/
assets/                 字体、图片、音频、视频、着色器
tests/                  仅保留未来重新建立的自动测试
tools/                  仅保留运行期或开发期工具

harness/                人工试玩记录、审核资料和后续验收设计
reference/              原版视频、截图、文档和不可执行参考资料
archive/                已核对的旧工程快照，不作为运行入口
```

“按软件层次”指先按职责划分，再在 `features/` 和 `content/levels/` 中按关卡区分；不再存在一整套手套工程、一整套头盔工程或一整套测试工程各自复制核心代码的结构。

## 唯一入口

- 正式运行入口：仓库根目录的 `project.godot`。
- `app/` 负责主入口和各关卡切换。
- 所有共享功能只允许从 `core/` 或 `gameplay/` 引用。
- 关卡专属逻辑不得复制 `GridWorld`、`WordEntity`、移动系统或视觉效果队列。
- `test game/`、根目录旧项目和 `剑流程/` 在迁移完成后不得再作为可交付运行入口。

## 迁移顺序

### 第一阶段：确定正式工程

最新合并已将手套关、头盔关、测试和共享核心实现同步到根目录。根目录因此是唯一可作为正式工程的候选，而不是 `newgame/`。第一步先修复从 `newgame/` 拷入根目录后仍指向 `res://../harness` 的外部路径，再确认根目录工程可以启动，之后才进行内部职责分层。

### 第二阶段：清理重复副本

逐文件比较 `test game/` 与根目录正式工程：

- 已完全相同的文件不再保留重复副本。
- `test game/` 独有的 `test_main.gd` 先迁入正式工程的测试或开发工具位置。
- 在对比清单确认后，将 `test game/` 移入 `archive/` 或删除其重复内容。

### 第三阶段：收拢旧根目录工程

先列出根目录现有旧场景、资源和第一章内容的职责归属，将它们迁入 `app/`、`core/`、`gameplay/`、`features/`、`content/` 和 `assets/`。根目录仍保留 `project.godot`，不再存在第二套根工程。

### 第四阶段：合入剑关

将 `剑流程/` 拆分为：

- 剑关流程与规则进入 `features/sword/`。
- 剑关地图和场景进入 `content/levels/sword/` 与 `content/scenes/`。
- 剑关音频、视频、字体和图片进入 `assets/`。

合入时必须直接使用正式工程的共享 `GridWorld` 和主入口，不得复制另一套核心系统。

### 第五阶段：内部职责分层

在唯一工程可启动后，按批次将现有 `scripts/`、`levels/`、`scenes/` 迁入目标目录。每个批次只移动一类职责，并同步更新该批次的 `res://` 引用、场景外部资源和 manifest 路径。

## 迁移约束

- 迁移期间不删除任何文件，直到完成来源、去向和重复性核对。
- 每次只迁移一个职责层或一个关卡包，禁止跨层混合大批量移动。
- 每次移动后必须由 Godot 导入并启动正式工程，确认没有脚本解析错误或资源路径错误。
- 自动验收当前仍暂停；这些启动检查只证明工程可加载，不代表玩法或像素复刻已验收。
- 所有新的开发只允许修改根目录正式工程，其他工程目录视为待迁移来源或归档。

## 成功标准

- 仓库根目录的 `project.godot` 是唯一正式运行入口。
- 共享核心代码每种职责只有一份正式实现。
- 手套、头盔和剑关均从同一正式工程加载。
- 每个移动批次均有文件映射清单和 Godot 启动记录。
- 旧副本没有未登记的独有脚本、资源或关卡数据。

## 非目标

- 不在本次迁移中修复玩法、地图、动画或手套关旧自动测试。
- 不把自动验收重新打开。
- 不通过删除目录来掩盖尚未合入的内容。
